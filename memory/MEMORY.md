## 📄 Действующие архитектурные решения (ADR)

### ADR-001: Выбор JWT вместо сессий
*   **Статус:** Принято (2026-03-10)
*   **Контекст:** Нам требуется распределённая stateless аутентификация для горизонтального масштабирования.
*   **Решение:** Использовать связку Access Token (JWT, время жизни 15 минут) и Refresh Token (хранится в защищённой HttpOnly cookie).
*   **Преимущества:** Сервисы могут валидировать Access Token локально по публичному ключу без походов в базу.

### ADR-002: Ротация Refresh Token (Refresh Token Rotation)
*   **Статус:** Внедрено (2026-05-24)
*   **Решение:** При каждом обновлении Access Token старый Refresh Token инвалидируется и выдаётся новый. Это защищает от повторного использования украденных токенов.

## ⚠️ Критические Спецификации и Баги

### ТЗ-089: Лимиты запросов (Rate Limiting) на /auth/login
*   **Статус:** В процессе (Алибек)
*   **Решение:** Ограничить количество попыток входа с одного IP (макс. 5 запросов в минуту). При превышении отдавать `429 Too Many Requests`.

### ТЗ-091: Баг инвалидации токенов при logout
*   **Статус:** Критическая ошибка (Prod)
*   **Проблема:** Refresh Token не удаляется корректно из базы/кэша сессий на бэкенде при вызове logout, что позволяет злоумышленнику использовать его повторно.
